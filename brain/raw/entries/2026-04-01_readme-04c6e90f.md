# Repository: affaan-m/everything-claude-code

**URL:** https://github.com/affaan-m/everything-claude-code

## 1. What It Does

Everything Claude Code (ECC) is a production-ready **agent harness performance system** — a curated, installable collection of prompts, agents, hooks, rules, commands, skills, and MCP configurations designed to turn Claude Code (and compatible harnesses like Codex, Cursor, OpenCode) into a high-output autonomous development environment. The core problem it solves is *context rot and capability degradation* in long-running Claude Code sessions: without structure, the model burns tokens on exploration, forgets patterns it discovered, repeats mistakes, and produces inconsistent output. ECC provides the scaffolding layer that keeps agents focused, memory-persistent, security-aware, and continuously self-improving. It is not just configuration — it ships a full continuous learning pipeline (instinct extraction → cluster → skill/command/agent evolution), a security scanning stack (AgentShield + InsAIts hooks), a deterministic harness scoring rubric, and a multi-agent orchestration system including git worktree parallelization. The project originated from an Anthropic hackathon win and represents 10+ months of daily production use building real products.

## 2. AI/Agent Architecture

### Orchestration Layer
The top-level control structure is Claude Code itself (the harness). ECC augments the harness rather than replacing it. Orchestration is expressed in three ways:

1. **Slash commands** (`/orchestrate`, `/multi-plan`, `/multi-execute`, `/loop-start`) — markdown files with YAML frontmatter that are injected into the Claude context as structured workflows. The `/orchestrate` command defines pipeline templates like `planner → tdd-guide → code-reviewer → security-reviewer` with explicit **handoff documents** between agents.

2. **tmux/worktree parallelism** — `scripts/orchestrate-worktrees.js` launches multiple git worktrees in separate tmux panes, each running an independent Claude instance. A `plan.json` file defines workers, tasks, and `seedPaths` (files overlaid from the main checkout). The `orchestration-status.js` script exports a live control-plane snapshot (tmux metadata, worker states, recent handoffs).

3. **PM2 integration** — for multi-service backend/frontend orchestration, `/pm2` command manages named processes; `/multi-backend` and `/multi-frontend` are specialized workflow commands for coordinating layered services.

### Agent Definitions
All 36 agents live in `agents/*.md` with YAML frontmatter:
```yaml
name: architect
description: "...Use PROACTIVELY when planning..."
tools: ["Read", "Grep", "Glob"]
model: opus
```
Tools are whitelisted per agent (e.g., `architect` gets read-only tools; build-resolvers get `Bash`). Model selection is explicit — high-judgment agents use `opus`, utility agents default to `haiku` or `sonnet`. The `description` field is how Claude Code's orchestrator decides *when* to invoke the agent.

### Skills as Prompt Modules
142 skills in `skills/*/SKILL.md` use markdown + YAML frontmatter as reusable prompt fragments. Skills are:
- **Curated** (shipped in `skills/`): authored, versioned, peer-reviewed
- **Generated** (saved to `~/.claude/skills/learned/`): auto-extracted by the `/learn` command at session end
- **Instinct-evolved** (stored in `~/.claude/homunculus/`): atomic learned behaviors with confidence scores, evolved from repeated session observations

The `continuous-learning-v2` skill (v2.1) implements a full **observe → cluster → evolve** pipeline. Hooks fire on every `PreToolUse`/`PostToolUse` event and capture observations asynchronously via a background Haiku sub-agent. Instincts are scoped per-project (git remote URL hash) to prevent cross-project contamination, with promotion to global scope when the same pattern is confirmed in 2+ projects.

### Hook Architecture
`hooks/hooks.json` is a declarative JSON file listing all lifecycle hooks. Hook phases:
- **PreToolUse** — validation, reminders, quality checks, security scanning, observation capture
- **PostToolUse** — formatting, type checks, build feedback, cost tracking, desktop notifications
- **Stop** — session summary persistence, session evaluation
- **PreCompact** — save important state before context window compaction
- **SessionStart** — load previous session state

Hooks are gated by `ECC_HOOK_PROFILE=minimal|standard|strict` and `ECC_DISABLED_HOOKS=...` env vars, allowing hot toggling without editing JSON. The `run-with-flags.js` wrapper reads these env vars at runtime before executing each hook script.

### Context Management
Context pressure is a first-class concern:
- **`/context-budget` skill**: inventories every loaded component (agents, skills, rules, MCP servers, CLAUDE.md files), estimates token consumption using `words × 1.3` heuristic, classifies components as `always-needed / sometimes-needed / rarely-needed`, and recommends lazy-loading or removal.
- **CLI flag injection**: `claude --system-prompt "$(cat memory.md)"` for surgical context loading — aliased as `claude-dev`, `claude-review`, `claude-research`.
- **Compact strategy**: strategic manual compaction at logical intervals via the `suggest-compact.js` PreToolUse hook, which fires on `Edit|Write` at defined intervals.
- **Session persistence**: Stop hook saves `.tmp` summary files in `.claude/`; SessionStart hook loads previous context automatically.

### MCP Integration
`.mcp.json` configures 6 MCP servers: `github` (PR/repo tools), `context7` (Upstash-backed context persistence), `exa` (web search over HTTP MCP), `memory` (Anthropic official memory server), `playwright` (browser automation), `sequential-thinking` (structured reasoning). The longform guide explicitly recommends *replacing CLI-wrapping MCPs with commands* to save context — e.g., `/gh-pr` wrapping `gh pr create` instead of loading the GitHub MCP constantly.

### Harness Scoring
`scripts/harness-audit.js` implements a deterministic 70-point scorecard across 7 categories: Tool Coverage, Context Efficiency, Quality Gates, Memory Persistence, Eval Coverage, Security Guardrails, Cost Efficiency. The rubric is version-stamped (`2026-03-30`) and file-check-based — reproducible for the same commit. Designed to audit both ECC itself and consumer projects using ECC.

## 3. Key Design Decisions

**1. Markdown-as-prompt-modules (not code).** Agents, skills, commands, and rules are all markdown with YAML frontmatter. This is intentional: LLMs read prose well, the format is self-documenting, and it enables rapid iteration without compilation. The tradeoff is no type safety or tooling — structure is enforced by CI tests (manifest validation, catalog counts, format checks).

**2. Hooks as the reliability backbone.** Rather than relying on the LLM to remember to do things (format code, check types, save session state), hooks enforce behaviors at the tool-use level unconditionally. Pre-commit quality (`pre-bash-commit-quality.js`) runs lint + secret detection *before* git commit regardless of whether Claude remembered. This inverts the trust model: the harness is authoritative, the LLM is cooperative.

**3. Instinct confidence scoring.** Learned behaviors are stored as YAML with a `confidence: 0.3–0.9` field derived from observation frequency and user corrections. Low-confidence instincts don't graduate to skills. This prevents the system from overfitting on one-off behaviors and avoids polluting the context with noise.

**4. Project-scoped memory isolation.** v2.1 adds per-project instinct storage keyed by git remote URL hash. React patterns don't leak into Python projects. Universal patterns (like "always validate input") are explicitly promoted to global scope. This solves a real production pain point where continuous learning systems contaminate unrelated projects.

**5. Cross-harness parity.** The same repo ships configs for Claude Code (`.claude/`), Cursor (`.cursor/`), Codex (`.codex/`), OpenCode (`.opencode/`), Kiro (`.kiro/`), Gemini (`.gemini/`), and CodeBuddy (`.codebuddy/`). Each harness gets a translated version of the same rules, skills, and hooks in its native format. This means the investment in prompt engineering compounds across tooling choices.

**6. Security as a hook, not an afterthought.** The `insaits-security-wrapper.js` hook runs on every `Bash|Write|Edit|MultiEdit` call. The `block-no-verify` hook prevents `git --no-verify` to protect pre-commit hooks from being bypassed. The security guide documents real CVEs (CVE-2025-59536, CVE-2026-21852) affecting Claude Code and explains prompt injection attack chains at depth.

**7. SQLite state store for session telemetry.** `scripts/lib/state-store.js` backs a SQLite DB tracking sessions, workers, skill runs, and decisions. `sessions-cli.js` provides a query interface. This enables structured retrospection on agent behavior across sessions.

## 6. Patterns Worth Borrowing

**1. Structured handoff documents between agents** (`commands/orchestrate.md`)
The `HANDOFF: agent-a → agent-b` markdown format with explicit Context/Findings/Files Modified/Open Questions sections is a clean, LLM-readable protocol for chaining agents. It prevents context loss between pipeline stages without requiring a shared memory system.

**2. Hook profile gating** (`scripts/hooks/run-with-flags.js`)
`ECC_HOOK_PROFILE=minimal|standard|strict` lets operators tune hook density at runtime without touching config files. Use this pattern in any system with optional enforcement layers — toggle by env var rather than code changes.

**3. Async background observation hook** (`hooks/hooks.json` + `continuous-learning-v2/hooks/observe.sh`)
Hook fires on every tool-use event with `"async": true, "timeout": 10` — observation runs in background Haiku without blocking the main context. This is the right pattern for any passive telemetry or learning layer: fire-and-forget, low cost, no latency impact.

**4. Instinct confidence + project scoping** (`skills/continuous-learning-v2/SKILL.md`)
Atomic instincts (one trigger → one action) with `confidence: 0.3–0.9` and project-hash scoping is a transferable design for any ML-light personalization system. The promote-to-global rule (seen in 2+ projects) is a principled generalization heuristic.

**5. System prompt aliasing for context mode switching** (`the-longform-guide.md`)
```bash
alias claude-dev='claude --system-prompt "$(cat ~/.claude/contexts/dev.md)"'
```
Surgical context injection via `--system-prompt` gives higher-authority context that's mode-specific. More practical than a bloated single CLAUDE.md.

**6. Deterministic harness scoring rubric** (`scripts/harness-audit.js`)
Seven categories, 0–10 normalized each, file-check-derived, version-stamped. Use this as a template for evaluating any AI-augmented codebase. The categories (Tool Coverage, Context Efficiency, Quality Gates, Memory Persistence, Eval Coverage, Security Guardrails, Cost Efficiency) map cleanly to any agentic system.

**7. seedPaths for worktree parallelism** (`commands/orchestrate.md`)
When launching parallel git worktrees, `seedPaths` overlays in-flight local files (plans, scripts) into isolated branches without committing them. Solves the "workers need to see my WIP" problem elegantly.

**8. MCP replacement with CLI commands** (`the-longform-guide.md`)
Replace CLI-wrapping MCPs (GitHub, Supabase, Vercel) with skill-backed slash commands to save ~500 tokens/tool in context overhead. Only load MCPs for capabilities with no CLI equivalent.

## 7. Gaps, Risks, and Limitations

**1. No programmatic guardrails on agent tool access.** Tool whitelisting (`tools: ["Read", "Grep", "Glob"]` in agent frontmatter) is a convention, not an enforced sandbox. A misconfigured or compromised agent definition could gain `Bash` access. Real sandboxing requires harness-level enforcement.

**2. Hook reliability is fragile at scale.** Hooks are Node.js scripts invoked by the harness process. If `CLAUDE_PLUGIN_ROOT` is unset, many hooks silently fail or fall back. The v1.9.0 changelog lists "observer loop prevention with 5-layer guard" and "memory explosion fix with throttling" — signs of hard-won stability, not inherent robustness.

**3. Instinct system has no eviction policy.** The homunculus instinct store grows monotonically. There's no TTL, staleness detection, or pruning logic documented. In long-lived projects, this could accumulate contradictory or outdated instincts that degrade rather than improve outputs.

**4. Cross-harness parity is manually maintained.** The same rules/skills/commands are duplicated into `.cursor/`, `.codex/`, `.opencode/`, `.kiro/` etc. with no automated sync. Drift between harness-specific configs is a real maintenance burden as the project scales (currently ~30 contributors).

**5. Security scanning is opt-in and external.** `insaits-security-wrapper.js` requires `ECC_ENABLE_INSAITS=1` and `pip install insa-its`. The AgentShield hook requires `ecc-agentshield` npm package. Neither is zero-config. Users who skip these steps have no runtime security layer beyond the rules markdown files.

**6. SQLite state store has no cloud sync.** Session telemetry lives in a local SQLite DB. There's no mechanism to aggregate session data across machines or teams, making the "continuous learning" story mostly per-developer. At team scale, patterns discovered by one developer don't propagate to others.

**7. Confidence scoring is heuristic-only.** Instinct confidence is incremented by observation count and user correction signals, but there's no ground-truth eval loop. A frequently-observed wrong pattern could gain high confidence. The `/learn-eval` command exists but is not deeply integrated into the confidence system.

**8. Markdown prompt modules have no formal schema.** Agents, skills, and commands use YAML frontmatter but the schema is loosely validated by CI tests rather than a JSON Schema or TypeScript type. This makes contribution and tooling harder than a structured format would allow.

## 8. How to Run It

### Prerequisites
- Node.js 18+ (for scripts and hooks)
- Claude Code installed and authenticated
- Git, bash (macOS/Linux) or PowerShell (Windows)
- Optional: Python 3 (for instinct CLI and InsAIts), tmux (for worktree parallelism)

### Environment Variables
```bash
cp .env.example .env
# Fill in:
ANTHROPIC_API_KEY=sk-ant-...
GITHUB_TOKEN=ghp_...
GITHUB_USER=your-username
DEFAULT_BASE_BRANCH=main
# Optional:
ECC_HOOK_PROFILE=minimal|standard|strict   # default: standard
ECC_DISABLED_HOOKS=pre:bash:tmux-reminder  # comma-separated
ECC_ENABLE_INSAITS=1                       # opt-in security monitor
```

### Install
```bash
git clone https://github.com/affaan-m/everything-claude-code.git
cd everything-claude-code
npm install

# Full profile (all components):
./install.sh --profile full

# Language-specific:
./install.sh typescript python golang

# Target a specific harness:
./install.sh --target cursor typescript
./install.sh --target gemini --profile full

# Windows:
.\install.ps1 --profile full
```

### Or via Claude Code plugin marketplace:
```
/plugin marketplace add affaan-m/everything-claude-code
/plugin install everything-claude-code@everything-claude-code
```

### Verify installation:
```bash
node scripts/harness-audit.js repo --format text
node tests/run-all.js
```

### Key runtime commands (inside Claude Code):
```
/harness-audit          # Score your setup
/context-budget         # Token inventory
/orchestrate feature "add auth"  # Run multi-agent pipeline
/learn                  # Extract session patterns
/instinct-status        # View learned behaviors
/loop-start             # Begin continuous eval loop
```
